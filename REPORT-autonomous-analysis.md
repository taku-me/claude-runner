# Claude Runner 自律分析レポート

**日付**: 2026-03-02
**対象リポジトリ**: [taku-me/claude-runner](https://github.com/taku-me/claude-runner)
**実行時間**: 約45分（30分 + 15分の2セッション）

---

## 1. 仕組み

### 概要

Claude Runner は「GitHub Issue を Claude Code に解かせる」自動化ツール。
今回の実験では、**課題の発見から修正まで** を完全自律で回した。

```
analyze.sh          run-task.sh            GitHub
    |                    |                    |
    | コードを読んで     |                    |
    | 改善点を発見       |                    |
    |                    |                    |
    |----- Issue 作成 ------------------>  Issue (#XX)
    |   (claude-task)    |                [claude-task]
    |                    |                    |
    |                    | Issue を検出        |
    |                    | コードを修正        |
    |                    | テスト実行          |
    |                    | commit & push      |
    |                    |                    |
    |                    |----- PR 作成 --->  PR (#YY)
    |                    | ラベル変更 -------> [claude-done]
    |                    |                    |
    +----- 次のサイクル --+                    |
```

### 3つのスクリプト

| スクリプト | 役割 | 入力 | 出力 |
|-----------|------|------|------|
| `analyze.sh` | リポジトリを読んで改善点を Issue として起票 | `--repo`, `--max-issues` | GitHub Issue（`claude-task` ラベル付き） |
| `run-task.sh` | 1つの Issue を読んで修正し PR を出す | `--repo`, `--issue` | GitHub PR + `claude-done` ラベル |
| `watcher.sh` | `claude-task` ラベルの Issue を監視して `run-task.sh` を呼ぶ | `--repo`, `--interval` | （デーモンとして常駐） |

### 自律ループの構成

今回のテストでは、以下のループを繰り返した:

```bash
while 時間内; do
  # 1. 分析: Claude がコードを読んで Issue を2件作成
  analyze.sh --repo taku-me/claude-runner --max-issues 2 --yes

  # 2. 実行: claude-task ラベルの Issue を順番に処理
  while claude-task が残っている; do
    run-task.sh --repo taku-me/claude-runner --issue $NUM
  done
done
```

**人間の介入: ゼロ。** `--yes` フラグにより Issue 作成の確認もスキップ。

## 3. Claude の特性分析

### 3.1 思考の順序: 「壊れるもの」から「美しくないもの」へ

目標を与えていないにもかかわらず、Claude は一貫した優先順位で動いた:

```
クラッシュするバグ  →  テストの穴  →  リファクタ  →  命名・型・慣例  →  ツール自体の堅牢性
     (高)                                                               (低)
```

これは人間のコードレビュアーの優先順位と同じ。「まず動くようにする、次に正しくする、最後に綺麗にする」という Kent Beck の格言に近い動きをしている。

### 3.2 視野の広がり: 対象コードからツール自身へ

最も興味深い特性。hello.py の改善ネタが枯渇すると、**自分を動かしている run-task.sh / watcher.sh / analyze.sh のバグ** を見つけ始めた。

時系列で見ると:

- サイクル1〜4: hello.py と test_hello.py に集中
- サイクル5: hello.py の型ヒント・PEP8（細かい改善 = ネタ切れの兆候）
- サイクル6: **analyze.sh と run-task.sh** に視野が移動
- サイクル7以降: **watcher.sh** の信頼性問題を発見

「自分自身の改善」を始めたのは指示されたからではなく、リポジトリ内のコードを読んだ結果として自然に発生した。

### 3.3 Issue の粒度: 一貫して小さい

全20件の Issue が「1関数の修正」「1行の追加」「1つの条件分岐の変更」レベル。大きなアーキテクチャ変更や複数ファイルにまたがるリファクタは一度も提案しなかった。

これは analyze.sh のプロンプトで「1つの小さな変更に絞る」と指示した効果もあるが、それ以上に Claude が **確実に完了できるサイズ** を選んでいる印象がある。結果として失敗率 0% を達成している。

### 3.4 重複回避: 効いているが完璧ではない

既存 Issue のタイトル一覧をプロンプトに含めることで、同じ問題を2回提案することは概ね防げた。ただし、意味的に近い Issue（#19 の None ガードと #25 の iterable 問題など）は別の Issue として出てくる。これは「同じ関数の別のバグ」なので重複とは言えないが、人間なら1つの Issue にまとめるかもしれない。

### 3.5 やらないこと

45分間で一度もやらなかったこと:

- **新機能の企画**: 「こういう機能があったら便利」とは考えない
- **大きなリファクタ**: ファイル構成の変更、アーキテクチャの刷新
- **主観的な判断**: 「このデザインパターンの方がいい」といった提案
- **破壊的変更**: 既存の挙動を変えるような修正

Claude は「今あるコードの問題を見つけて直す」に徹しており、**創造的な意思決定** は行わない。これは制約であると同時に、自律運用での安全性を担保している。

## 4. 実用性の評価

### うまくいったこと

- **完全自律で45分間、失敗ゼロ**: Issue 作成 → 修正 → PR 作成のパイプラインが安定して動いた
- **実際に価値のある修正**: #44（PR重複防止）、#49（失敗時の再試行）などはツール自体の実運用で遭遇するバグ
- **テスト確認付き**: 全 PR で既存テストの通過を確認してからコミットしている

### 課題・注意点

- **main にマージしないと次のサイクルに反映されない**: 各 PR は独立ブランチなので、サイクルが進んでも main のコードは変わらない。同じ箇所を別の観点で修正する PR が競合する可能性がある
- **小さいリポジトリ向き**: hello.py + シェルスクリプト数本のリポジトリでは45分で改善ネタが枯渇気味。大きなリポジトリではもっと長く有意義に動けるはず
- **品質の判断は人間が必要**: #36（greet_all → hello_all リネーム）のような命名変更は、チームの合意なしに入れるべきではない。自律で PR を作るところまでは良いが、マージの判断は人間がすべき

## 5. まとめ

Claude に目標を与えず「このリポジトリを改善して」とだけ言うと、**保守的で体系的なコードレビュアー** として振る舞う。壊れるバグから順に潰し、テストを足し、コード品質を上げ、最終的にはツール自身の信頼性まで改善する。

これは「自律的に開発を進める AI」というよりも、**「無限に働くジュニアレビュアー」** に近い。創造性や判断力を求めるタスクには向かないが、「見つけて直す」の繰り返しには極めて有効。

人間がやるべきことは、PR をレビューしてマージするかどうかを決めること——つまり **承認権限を持つシニアエンジニア** の役割に集中できる。
